===========================================================================
  AI Terminal Execution & Development Guidelines For LLM Chat Interfaces
===========================================================================

1. WORKFLOW RULES (CRITICAL)
----------------------------
1.1  Terminal Only
     Do not ask the user to manually create or modify files. Provide full 
     terminal commands using `cat << 'EOF' > path/to/file` to create or 
     overwrite files. Organize terminal commands into logical, clean batches.

1.2  No Placeholders
     Write complete, fully functional, compilation-safe code. Avoid using 
     comments like "// ... rest of code" or placeholder stubs.

1.3  Step-by-Step Phase-Based Workflow
     Before making changes, outline the phases of work. Provide the files and 
     patches sequentially in distinct, logical steps rather than all at once.

1.4  Verification Checklist
     After providing code, list specific, actionable manual tests or compile 
     checks for the user to run to verify the change has succeeded.

1.5  Atomic Commits
     When requested, output Git commit command sequences. Format as:
       git add . && git commit -m "title" -m "description" && git push
     Do not present commit commands until explicitly requested. Always use the 
     period (.) for `git add` to capture all staged modifications.

1.6  Generative Procedural
     Generate game assets, procedural waveforms, and canvas graphics programmatically.

1.7  WAIT (Communication Checkpoint)
     Do not output extensive code blocks for a proposed plan until the user 
     confirms the outline. Articulate the approach clearly first, then wait 
     for the confirmation signal.

1.8  Local Testing
     All verification must be supported by running the local Vite server 
     development commands (`npm run dev`) and verifying build stability 
     (`npm run build`).

1.9  Line Diagnostics for Debugging
     When debugging or analyzing an error/crash stack trace where the exact 
     line contents of a file are uncertain, always run a quick temporary 
     diagnostic script (or evaluation command) to inspect and output the 
     relevant lines before designing a patch or overwrite.


2. CODE STANDARDS & TERMINAL PROTOCOLS
--------------------------------------
2.1  Node Scripts Must Be ESM
     Since this project operates with "type": "module" in package.json, all 
     temporary or auxiliary Node.js scripts must use ES Module syntax 
     (e.g., use "import fs from 'fs';" instead of "const fs = require('fs');").

2.2  Surgical Patch Scripts
     When modifying large source files, prefer writing temporary Node.js 
     scripts (utilizing `fs.readFileSync` and `.replace()`) over full file 
     overwrites to prevent command truncation or buffer loss.
     CRITICAL: Always append the cleanup command (e.g., "&& rm ./patch.js") 
     on the exact same execution line to keep the repository untracked space clean.
     Actively clean up and delete any other transient scripts or patch helpers 
     immediately after execution to prevent workspace clutter.

2.3  Zsh Compatibility & Escaping
     The development environment is on macOS running Zsh. Never use raw, unescaped 
     exclamation marks (!) inside shell string variables, as Zsh parses them 
     as history expansion operators and will fail execution. 

2.4  Zero Bash Comments
     Never include hash symbols (#) inside any active terminal or shell code 
     blocks. Copy-pasting shell commands containing hash lines directly into 
     macOS interactive Zsh terminals triggers immediate parsing errors or partial 
     execution. Keep shell blocks strictly functional and describe operations 
     in the accompanying markdown text instead.

2.5  Strict Structured File Patching
     When modifying CSS, JSON, or complex TypeScript layouts with nested brackets, 
     do not use lazy regex matching (e.g. "[\s\S]*?") to replace large blocks, 
     as it easily misaligns closing braces and corrupts files. Perform full 
     overwrites for small-to-medium files or strict literal string matching 
     for surgical updates.

2.6  Unicode and Escapes in Bash
     When writing inline scripts via terminal heredoc, never use raw unicode 
     escapes (such as "\u00A0") if they risk being parsed by the terminal shell 
     as literal invisible characters. Instead, use runtime code conversions 
     like `String.fromCharCode(160)` to safely inject special characters.

2.7  Quoted Starting Delimiters (Shell Protection)
     Always wrap the starting heredoc delimiter in single quotes (e.g., 
     `cat << 'EOF' > file` or `cat << 'ENDOFSCRIPT'`). Unquoted delimiters 
     allow the shell to perform immediate variable expansion ($variable) and 
     backtick execution before writing, stripping or corrupting raw JS, TS, 
     and CSS structures.

2.8  Delimiter Syntax and Formatting
     The terminating delimiter (e.g., EOF or ENDOFSCRIPT) must sit alone on its 
     own line with zero leading or trailing whitespace. Any leading spaces or 
     tabs will prevent the terminal from recognizing the block boundary, 
     causing terminal lockup or truncation.

2.9  Strict TypeScript Constraints
     Always comply with strict compiler rules (such as 'noUnusedLocals', 'noUnusedParameters',
     and '@typescript-eslint/no-explicit-any') by stripping unused variables, constants, or imports,
     and completely avoiding explicit 'any' type declarations. If a general or dynamically resolved
     type is required, store references as 'unknown' and perform type assertions, narrow type guards,
     or implement clean generic abstractions to ensure type-safe interfaces. When modifying or expanding
     strongly typed configurations, nested structures, or read-only literals (e.g., 'as const'), verify
     that all symmetrically accessed keys exist across all related record types or union branches to
     prevent compile-time property errors (TS2339).


3. REMOTE INSPECTION & EDITING TECHNIQUES
-----------------------------------------
When debugging, analyzing, or modifying files, select the most efficient 
protocol based on scope:

3.1  Quick Inline Checks (node -e)
     For single-purpose evaluations (verifying file contents, counting pattern 
     instances, checking directory listings):
       node -e "console.log(require('fs').readFileSync('src/App.tsx','utf8').includes('useGameStore'))"

3.2  Multi-Line Analysis Scripts (Heredoc + Self-Delete)
     For complex log analysis, directory traversal, or file verification:
       cat << 'ENDOFSCRIPT' > ./analysis.js
       import fs from 'fs';
       import path from 'path';
       // Analysis logic...
       ENDOFSCRIPT
       node ./analysis.js && rm ./analysis.js

3.3  File Modification by Full Overwrite
     For small to medium files, overwrite the target path directly:
       cat << 'EOF' > src/components/ui/MyNewComponent.tsx
       // Complete implementation...
       EOF

3.4  File Modification by Surgical Patch
     For large files where only specific lines need adjustment:
       cat << 'ENDOFSCRIPT' > ./patch.js
       import fs from 'fs';
       let content = fs.readFileSync('src/components/GameArena.tsx', 'utf8');
       content = content.replace('old code block', 'new code block');
       fs.writeFileSync('src/components/GameArena.tsx', content);
       ENDOFSCRIPT
       node ./patch.js && rm ./patch.js

3.5  Build Verification
     Validate all TypeScript compilations and bundler configurations after any 
     file updates before concluding:
       npm run build

    const shuffleArray = (array) => {
      const arr = [...array];
      for (let i = arr.length - 1; i > 0; i--) {
        const j = Math.floor(Math.random() * (i + 1));
        [arr[i], arr[j]] = [arr[j], arr[i]];
      }
      return arr;
    };

    function getShuffledIndices(length) {
      return shuffleArray(Array.from({ length }, (_, i) => i));

    function GlobalScrollbar({ lenisInstance }) {
      const [scrollY, setScrollY] = useState(0);

        window.addEventListener('resize', () => {
          handleScroll();

        });
        handleScroll();
        measureHeader();
        const timer = setTimeout(measureHeader, 100);
        return () => {
          window.removeEventListener('scroll', handleScroll);
          window.removeEventListener('resize', handleScroll);
          clearTimeout(timer);
        };
      }, [])

      const progress = maxScrollY > 0 ? scrollY / maxScrollY : 0;

        if (!trackRef.current) return;
        const rect = trackRef.current.getBoundingClientRect();
        const clickY = e.clientY - rect.top;
        const ratio = Math.max(0, Math.min(1, clickY / rect.height));
        const targetY = ratio * maxScrollY;
        
        if (lenisInstance) {
          lenisInstance.scrollTo(targetY);
        } else {
          window.scrollTo({ top: targetY, behavior: 'smooth' });
        }
      };

      const handlePointerDown = (e) => {
        e.preventDefault();
        document.body.style.userSelect = 'none';

        const startY = e.clientY;
        const startScroll = window.scrollY;

        const handlePointerMove = (moveEvent) => {
          if (!trackRef.current) return;
          const deltaY = moveEvent.clientY - startY;
          const trackHeight = trackRef.current.clientHeight;
          const scrollDelta = (deltaY / trackHeight) * maxScrollY;
          const targetY = Math.max(0, Math.min(maxScrollY, startScroll + scrollDelta));
          
          if (lenisInstance) {
            lenisInstance.scrollTo(targetY, { immediate: true });
          } else {
            window.scrollTo(0, targetY);
          }
        };

        const handlePointerUp = () => {
          document.body.style.userSelect = '';
          document.removeEventListener('pointermove', handlePointerMove);
          document.removeEventListener('pointerup', handlePointerUp);
        };

        document.addEventListener('pointermove', handlePointerMove);
        document.addEventListener('pointerup', handlePointerUp);
      };
